HOW VELOCITY IS BUILT & RUN

Operations Playbook

The operating manual for Velocity: where everything lives, how code reaches production, how the database moves with it, how to QA, and what to do when something breaks. If you're shipping a change, start at "Branch & deploy flow" and "Database & migrations".

1 · Environments

Code · hosting · database · DNS

Velocity is one codebase deployed to two environments. Hosting is Railway (auto-deploy from branches); the database is a single Supabase project with two branches; DNS is Cloudflare.

Both Supabase branches live under the one project velocity (org CoreShift). They are separate databases — data and schema do not sync automatically. A migration applied to staging is not on prod until you apply it there too (see §3).

Health check: GET /health returns uptime — a low uptime confirms a fresh deploy landed.

2 · Branch & deploy flow

local → dev → staging → main
local→ dev→ staging→ main (prod)
  • Work locally, commit on a feature/work branch.
  • Push to staging → Railway auto-deploys the staging environment. (Small changes push straight to staging; the dev branch + GitHub Actions build-gate is the path for larger PR-reviewed work.)
  • Promote staging → main by PR → Railway auto-deploys production.
Rules. Never push directly to main — always go through staging. Production promotion needs Abe's sign-off. CI runs tsc + build as a hard gate; it does not run DB migrations (see §3).

Before any ship: npx tsc --noEmit and npm run build must both be green.

3 · Database & migrations

Drizzle + Supabase branches

Schema lives in shared/schema.ts. Migrations are SQL files in migrations/NNNN_*.sql with a matching entry in migrations/meta/_journal.json.

Writing a migration

  • Add the column/table to shared/schema.ts (+ an insert…Schema export if needed).
  • Create migrations/00NN_name.sql — make it idempotent (CREATE TABLE IF NOT EXISTS, ADD COLUMN IF NOT EXISTS, CREATE INDEX IF NOT EXISTS) so it's safe to re-run.
  • Append the entry to _journal.json.

Applying it

Deploys do not auto-run migrations. Apply the SQL to each Supabase branch out-of-band (idempotent SQL via the Supabase MCP / SQL editor), in step with the code:

WhenWhere
Ship to stagingApply to the staging branch pdawstndulfnydvesmdm

The #1 thing not to forget: when promoting to production, apply the new migrations to the prod (main) Supabase branch before/with the code deploy. If the app deploys expecting a table/column the prod DB doesn't have, requests 500. Current batch awaiting prod: 0011 shared facts, 0012 source_field, 0013 notifications.

4 · How to QA

Verify on staging before promoting
  1. Confirm the deploy landed: GET /health shows a fresh (low) uptime, and the served JS bundle contains the new strings.
  2. Probe new routes unauthenticated — expect 401 (auth-guarded & live), not 404 (missing route).
  3. Exercise the change in the app on velocity-staging.coreshifthq.com with a test deal (e.g. "Abe Test 3").
  4. For gated flows, check the gate progress card + the server actually enforces it (a forward drag opens the checklist; an unmet forward move is refused).
  5. For buyer-facing changes, open the deal's portal via its magic link in a separate/incognito tab.
The extractor and AI features run in real mode only when ANTHROPIC_API_KEY is set on the environment; otherwise they fall back to a labelled "stub" mode (keyword-only).

5 · Production promotion checklist

staging → main
  • tsc + build green on the staging head commit.
  • Staging QA passed for everything in the batch.
  • List the migrations in the batch (diff migrations/ since the last prod release).
  • Get Abe's sign-off to promote.
  • Apply each new migration's idempotent SQL to the prod Supabase branch (uodsmyystkaxoogitvsw).
  • Open + merge the PR staging → main (gated).
  • Wait for Railway prod deploy; confirm GET /health uptime reset.
  • Smoke-test prod: a probe route returns 401, the bundle has the new code, a key page loads without 500.
  • Update the build board (move cards from "on staging" → "in prod").

6 · Connectors

Google OAuth · per-user tokens

Google Calendar uses a single OAuth client. Credentials live in Railway env vars on both environments: GOOGLE_OAUTH_CLIENT_ID, GOOGLE_OAUTH_CLIENT_SECRET, GOOGLE_OAUTH_REDIRECT_URI_GCAL. Tokens are stored per-user in user_connectors; connectorTokens.ts auto-refreshes and flips a connector to expired on invalid_grant.

  • Each environment's redirect URI must be registered on the Google OAuth client (the callback can only be added by an operator with GCP access).
  • The Calendar API must be enabled in the GCP project (operator-only).
  • Settings only shows a live "Connect" for built connectors (an implemented flag); others read "Coming soon".

7 · Incidents & rollback

When prod misbehaves
  1. Confirm scope: GET /health, then railway logs for the prod service.
  2. If it's a bad deploy: roll back by reverting the offending commit on main (CI redeploys), or redeploy the previous good commit in Railway. Prefer revert over force-push.
  3. If it's a missing migration (most likely cause of a fresh-deploy 500): apply the migration's idempotent SQL to the prod branch — that usually resolves it without a code rollback.
  4. Never run destructive DB operations on prod to "fix" an incident; additive/idempotent only.
Railway is used for logs + env vars only — never deploy manually; CI handles deploys from the branches.
